# CLAUDE.md

## プロジェクト概要
個人発・公開型の「雑学・知識データベースWebアプリ」。
アプリ名は TriviaVerse。
自分で調べた雑学や知識を登録・整理し、キーワードやカテゴリ（タグ）で
検索・閲覧できる知識辞典。自分は読み書きでき、他人は読むことができる。
PC・スマホ両方から同じURLで使える（公開済み）。
（フォルダ名・プロジェクト名は trivia-app）

## ターゲットユーザー
- 主：開発者本人（情報系の大学生）＝唯一の書き込み者
- 副：閲覧者（他人。読むことのみ可能）
- 開発者のスキル：プログラミングはほぼ初心者

## 解決する課題
新たな知識・雑学を増やしたいが「探す・理解する・続ける」がめんどうで定着しない。
→ 今回のスコープは「自分で調べた知識を散らからず集めて整理し、後から探せる
   状態にし、必要なら他人にも見せられる」ことに集中。
   （きっかけ作り＝再表示・通知は将来の課題として保留）

## 主要機能
- 【登録】タイトル → 本文 の順で入力して知識を追加
- 【本文2分割】AIの説明（貼り付け）と 自分のまとめ（自分の言葉）を分けて保存
  - 少なくともどちらか一方が入力されていれば登録可
- 【タグ分類】タグで分類。複数付与可・0個可・後から編集可
- 【画像】知識に画像を1枚添えられる（任意）。詳細画面で表示され、理解の助けになる
  （画像は Supabase Storage に保存し、URLを knowledge.image_url に持つ。詳細は下記）
- 【コード＋解説】知識にプログラムコードと、その処理の説明を添えられる（任意）。
  詳細画面で「コード（色付き）」と「解説」を隣り合わせ（PCは左右／スマホは上下）で表示する。
  プログラミングの学びを、コードと説明をひと目で対応させながら貯められる（詳細は下記）。
- 【検索・閲覧】キーワード検索＋タグ絞り込み（併用可）
- 【閲覧】詳細画面で本文・出典・タグ・画像・コード＋解説を表示（入力時の改行はそのまま表示）
- 【編集・削除】後から修正・削除（CRUD）。画像も差し替え・削除できる
- 【権限】自分＝読み書き／他人＝読み取りのみ
- 【ホームは今週分】ホーム一覧は「今週（木曜0:00〜翌週水曜23:59）に追加した知識」だけを表示。
  「すべて見る」ボタンで全件表示に切り替え可能（→ 改善案3後半）
- 【ランダム閲覧】登録済みからランダムに最大10件を表示する画面（→ 改善案4）

## 画面構成（1ページ内で切り替える方式）
- index.html の中に、共通ヘッダー＋4つの画面エリアを持つ。
  - #view-home …… ホーム（検索・絞り込み＋登録済みの知識の一覧。既定は今週分）
  - #view-detail … 詳細（1件をじっくり表示。ログイン中は編集・削除ボタン）
  - #view-edit …… 登録／編集フォーム（画像の選択・プレビュー・削除チェック、
    コード／言語選択／コード説明の入力欄を含む）
  - #view-random … ランダム閲覧（最大10件をランダム表示。未ログインでも使える）
- app.js の showView("home" / "detail" / "edit" / "random") で表示を出し分ける
  （実際のページ遷移ではなく、1ページ内で見せる部分を切り替えるSPA的な方式）。
- 一覧・ランダムの各カードのタイトルをクリック → 詳細画面へ。
- ログインは普段は隠し、「ログイン」ボタン(#login-toggle)を押したときだけ表示。
- 「＋ 新規登録」ボタンはログイン中のみ表示。

## データ構造（knowledgeテーブル）
- id：int8・主キー・自動採番（Is Identity）
- created_at：timestamptz・now()（自動）
- title：text（必須）
- ai_explanation：text（任意）　┐ どちらか一方は必須（アプリ側でチェック）
- my_summary：text（任意）　　 ┘
- source：text（任意）
- tags：text[]（配列・任意・複数可）
- image_url：text（任意・Nullable）。添えた画像の「公開URL」を保存する。
  画像ファイル本体ではなくURLの文字列だけが入る（本体は Storage に保存）。
- code：text（任意・Nullable）。プログラムコード本体を保存する。
- code_explanation：text（任意・Nullable）。コードの処理を説明する文章を保存する。
- code_language：text（任意・Nullable）。言語名（python / javascript など）を保存する。
  highlight.js が認識する言語名を入れる（色付けに使う。例 C++ は "cpp"、C# は "csharp"）。

## 画像の保存方式（Supabase Storage を使う＝方式A）
- 画像ファイルは Supabase Storage の公開バケット「knowledge-images」に保存する。
  アップロードで得た公開URLを knowledge.image_url に記録し、表示時はそのURLを <img> で読む。
- なぜこの方式か（設計判断の記録）：
  - 方式B（URLだけ持つ）も検討したが、容量と向き合いたい・外部に依存せず自己完結
    したい、という理由で方式A（ファイル本体を自分の Storage に持つ）を選んだ。
  - 方式Aは「アプリ内で画像をアップロードできる」体験が得られ、リンク切れもない。
- 容量の考え方：Storage 無料枠は約1GB。画像1枚 数百KB〜数MB なので、当面（数千枚規模まで）
  は問題なし。将来溜まってきたら、アップロード前の縮小やサイズ上限の設定を検討する。
- アップロードの仕組み（app.js）：
  - db.storage.from("knowledge-images").upload(path, file) でアップロード。
  - path は衝突回避のため必ずユニークにする
    （Date.now() + "_" + ランダム文字 + "." + 拡張子。日本語ファイル名でも安全）。
  - db.storage.from("knowledge-images").getPublicUrl(path) で公開URLを取得し image_url に保存。
  - 画像なしで登録も可（image_url は null）。編集時に選び直さなければ既存画像を維持、
    「画像を削除する」チェックで image_url を null にする（editingItem 変数で旧値を保持）。

## コード＋解説機能（設計判断の記録）
- 何を：知識にプログラムコードと、その処理の説明（解説）を添え、詳細画面で
  「コード（左）＋解説（右）」を隣り合わせに見せる。PCは左右2カラム、スマホは上下1カラム。
- 設計判断：
  - 構成は「シンプル案」を採用＝コード1つ＋説明1つを左右に並べる。
    （行ごと・ブロックごとに説明を結びつける「リッチ案」は複雑なので将来検討）。
  - 色付け（シンタックスハイライト）は最初から入れる。ライブラリは highlight.js を採用。
  - 言語は code_language 列に「データとして持つ」方式を採用（自動判定ではなく明示）。
    表記ゆれを防ぎ色付けを安定させるため、登録時は固定のドロップダウンから言語を選ぶ。
- 色付けライブラリ（highlight.js）：
  - index.html で CDN から読み込む（テーマCSS = atom-one-dark、本体JS）。
    読み込み順は app.js より前（supabase-js と同じ並び）。
  - 詳細画面でコード要素を作った後に hljs.highlightElement(その要素) を呼んで色付けする
    （自動の highlightAll ではなく該当要素だけを明示的に。window.hljs の存在チェック付き）。
- 表示の仕組み（app.js の showDetail）：
  - item.code があるときだけ、コード＋解説のブロック（.code-block-wrap）を作る（無ければ出さない）。
  - コードは <pre><code class="language-XXX"> に入れる（XXX=code_language）。
  - 重要：コードは必ず textContent で挿入する（innerHTML は使わない）。
    コード中の < > & がHTMLとして誤解釈される事故や XSS を防ぐための安全策。
  - 解説は white-space: pre-wrap で改行を保持して表示する。
- 言語ドロップダウンの選択肢（value は highlight.js 認識名）：
  指定なし(空), python, javascript, typescript, java, c, cpp, csharp, html, css, sql, bash, json。

## デザイン方針（段階3で基礎、改善案2でホームズ風に発展）
- テーマ：シャーロック・ホームズ風の「知的な書斎」。
  ヴィクトリア朝の書斎をイメージした、落ち着いて知的で少し古風な雰囲気。
- 「眺める・出会う楽しさ（アプリを開くこと自体が楽しい）」を最重要に置いてデザインした。
  （書くときの演出や件数のお祝いは今回は作らない、と方針決め）
- 配色（style.css の :root に CSS変数でまとめ、後から調整しやすくしている）：
  - --paper（淡い羊皮紙色）＝ページ全体の背景
  - --paper-card（明るいクリーム）＝コンテンツ領域
  - --navy（深い紺）＝見出し・主要テキスト・主ボタン
  - --gold / --gold-dark（くすんだ真鍮ゴールド）＝アクセント（見出し下線・枠・ホバー）
  - --border / --line（温かみのあるベージュ）＝罫線・区切り
  - --text（温かみのある濃い茶）＝本文
  - ※以前の --base-bg / --card-bg は別名として残し後方互換を確保
- フォント：見出し（h1/h2/h3）はセリフ体（--serif）で格調を出し、
  本文・入力欄・ボタンは読みやすいゴシックのまま（可読性と格調の両立）。
- タグの色分け：タグ名から自動で色を決める（同じタグは常に同じ色）。
  app.js の tagColor() がタグ名の文字コード合計をパレット数で割った余りで色を選ぶ。
  パレットは書斎に合う深く落ち着いた8色（深緑・ワイン・濃紺・マスタード・こげ茶・
  青磁・深い紫・テラコッタ）。タグ絞り込みボタンは背景を薄く・文字と枠を濃く（上品に）。
- カード（本の背表紙イメージ）：
  - 一覧・ランダムの各カードは左端に「最初のタグの色」の縦ライン（背表紙風。
    タグが無いカードは控えめな金）。app.js の spineColor() が色を決める。
  - マウスホバーで少し浮き上がる（影＋translateY）。タイトルは金色に変わる。
  - カードのタイトルはセリフ体。
- 一覧の並べ方：CSS Grid で書棚のようにタイル状（横に複数枚）に並べる。
  grid-template-columns: repeat(auto-fill, minmax(220px, 1fr)) で画面幅に応じ枚数自動調整。
  ホーム一覧・ランダムの両方に適用。
- ランダムの演出：カードが1枚ずつ少し遅れてふわっと現れる（cardFadeIn ＋ .card-appear、
  animation-delay を i*0.05s で付与）。ランダム画面だけに付け、ホーム一覧には付けない
  （「引き直す楽しさ」を出すため）。
- 詳細画面：AIの説明＝羊皮紙色＋金茶の枠、自分のまとめ＝紺がかったクリーム＋紺の枠、
  それぞれバッジ付きで視覚的に区別（→ 改善案1）。
  本文は white-space: pre-wrap で、入力時の改行をそのまま表示する（→ 使い勝手の改善）。
  画像があればタイトルの下に表示（.detail-image。最大幅100%・角丸・ベージュ枠で馴染ませる）。
- 詳細画面のコード＋解説（.code-block-wrap）：
  - PCは左右2カラム（grid-template-columns: 1fr 1fr）、スマホ（600px以下）は上下1カラムに切替。
  - コードの枠（pre）は暗色背景（#282c34＝atom-one-dark）＋等幅フォント＋角丸、
    長い行は overflow-x:auto で横スクロール（折り返さない）。
  - 「コード」バッジ（紺）／「解説」バッジ（金茶）で区別。既存 field-badge の雰囲気に合わせる。
- 入力欄（textarea）：AIの説明・自分のまとめは初期 rows=6 で広めに表示し、
  resize: vertical で縦方向に伸縮できる（長文を書きやすく。→ 使い勝手の改善）。
  コード・コード説明の欄は rows=8。
- ボタン：主ボタン（ログイン/登録/検索）は濃紺＋白文字＋金の縁、
  副ボタンはクリーム＋紺枠でホバーに金を効かせる。
- スマホ対応：画面幅600px以下で余白・文字を調整し、検索欄とボタンを縦積みに。
  一覧・ランダムのグリッドは1列に戻す。コード＋解説も1カラムに積む。

## セキュリティ設定（実施済み）
- テーブル(knowledge)の RLS有効。ポリシー2つを設定済み：
  - 「Enable read access for all users」：SELECT / public（誰でも読み取り可）
  - 「Allow authenticated write」：ALL / authenticated（ログイン者のみ書き込み可）
- Storage（knowledge-images バケット）のポリシー2つを設定済み：
  - 「Public read for knowledge-images」：SELECT / public（誰でも画像を読める＝表示できる）
  - 「Authenticated insert for knowledge-images」：INSERT / authenticated
    （ログイン者だけが画像をアップロードできる）
  - 考え方はテーブルの RLS と同じ：読み取りは誰でも、書き込みはログイン者だけ。
  - バケットは Public（URLを知っていれば画像を閲覧可）。人に見られて困る画像は置かない前提。
- 認証：メール＋パスワード方式。自分のユーザーはSupabase管理画面で作成済み
- Data API：public スキーマ公開、knowledge テーブルを個別に公開済み
- コードの安全表示：詳細画面でコードは textContent で挿入（innerHTML を使わない）。
  コード中の < > & の誤解釈や XSS を防ぐ。
- 守っている習慣：
  - 秘密キーはコードに置かない（公開キーのみ使用）
  - 公開前に config.js の中身を必ず目視確認
  - 外部サービス連携は最小権限（Vercel の GitHub アクセスは「選択したリポジトリのみ」）
  - キー類は公開予定のものでも、画面共有時は隠す癖をつける
  - 認証・キー入力・ポリシー設定など重要操作は必ず本人の手で行う（AIに代行させない）

## 公開作業の記録（段階3・学びの備忘録）
- config.js は公開キー（sb_publishable_）のみで秘密情報を含まないため、
  .gitignore から外して公開対象にした（方式B）。
- 当初リポジトリを Private で作成 → Vercel無料(Hobby)プランでは
  Privateリポジトリのデプロイに制約があり、2回目のデプロイが Blocked になった。
- 解決：GitHubリポジトリを Public に変更 → 新しいコミットを push して再デプロイ成功。
  （config.js の中身が原因ではなく、「Private × 無料プラン」の制約が原因だった）
- GitHub認証は GitHub CLI（gh）で実施（brew install gh → gh auth login）。
- 教訓：無料で個人開発を公開するなら、最初からリポジトリを Public にしておくと
  デプロイの制約に引っかからずスムーズ（公開して困る秘密情報を置かない前提）。
- push 前に必ず git status で「何が変わるか」を確認する習慣を実践。

## Storage 導入の記録（画像機能・学びの備忘録）
- 画像機能の追加では、アプリのコードを書く前に Supabase 側の下準備を3段階で行った：
  1. Storage に公開バケット「knowledge-images」を作成（サイズ制限・MIME制限はオフ＝後で足せる）。
  2. knowledge テーブルに image_url 列（text, Nullable）を追加。
  3. Storage のポリシーを2つ作成（誰でも読める SELECT / ログイン者だけの INSERT）。
- 学び：Storage にもテーブルの RLS と同じ「ポリシー」の考え方がある。
  これを設定しないと、アプリからアップロードしようとしても拒否される。
  「読み取りは公開、書き込みはログイン者だけ」を、テーブルと Storage の両方で揃えるのが基本。
- 進め方：新しい仕組み（Storage）は、いきなり全部やらず「バケット→列→ポリシー→アプリ実装」
  と小さく刻み、各段階で画面を確認しながら進めた。

## 外部ライブラリ導入の記録（コード機能・学びの備忘録）
- コード機能では、色付けに highlight.js という外部ライブラリを CDN から読み込んで使った。
  （supabase-js と同じく、index.html に読み込みタグを足すだけで使える）
- 進め方：画像機能と同じく「テーブルに列を追加（code/code_explanation/code_language）→
  アプリ実装」と段階を分け、各段でローカル確認 → push。
- 学び・つまずきポイント：
  - コードは textContent で入れる（innerHTML は禁止）。コードには < > & がよく出るため、
    HTMLとして誤解釈されないようにする必要がある（安全＆表示崩れ防止）。
  - 色付けを安定させるには言語の指定が要る。自動判定もあるが、短いコードだと外すため、
    ドロップダウンで言語をデータとして持たせる方式にした（表記ゆれも防げる）。
  - 「PCで左右・スマホで上下」は CSS Grid の grid-template-columns を
    メディアクエリで 1fr 1fr → 1fr に切り替えて実現（案2のグリッドと同じ考え方）。

### 案1：AIの説明と自分のまとめを視覚的に分ける（✅ 完了）
- 何を：詳細画面で「AIの説明」と「自分のまとめ」を、色付きの枠＋バッジで
  ひと目で区別できるようにした。一覧はタイトルのみのスッキリ表示に。
- 実装：詳細画面で AI=羊皮紙色＋金茶枠（.field-ai）、自分=紺系の枠（.field-mine）、
  それぞれバッジ（.field-badge）付き。app.js の showDetail が組み立てる。

### 案2：眺める・出会う楽しさのデザイン（✅ 完了／ホームズ風）
- 何を：アプリを開くこと自体が楽しくなる見た目に。テイストは
  「シャーロック・ホームズ風の知的な書斎」に決定。
- 実装（詳細は「デザイン方針」参照）：
  - 配色を羊皮紙クリーム＋紺＋真鍮ゴールドに。見出しをセリフ体に。
  - タグを自動で色分け（tagColor）。同じタグは常に同じ色。
  - カード左端にタグ色の背表紙ライン（spineColor）＋ホバーで浮き上がり。
  - 一覧・ランダムを CSS Grid で書棚風のタイル表示に。
  - ランダムはカードが順にふわっと現れる登場アニメーション（cardFadeIn）。
- 進め方メモ：まず「芯」（配色・タグ色・カード・セリフ体）を入れて確認 → push、
  続いて「追加分」（グリッド・登場アニメ）を入れて確認 → push、と2段階で実施。

### 案3：登録数と「今週の知識」を分かりやすく（✅ 完了）
- 前半（件数表示）：updateCountStatus が全体件数を取得し、状況に応じて
  「全◯件」「◯件見つかりました（全△件中）」「今週◯件（全△件中）」を出し分け。
- 後半（ホームを今週分に）：ホーム一覧は既定で「今週（木曜0:00〜翌週水曜23:59）に
  追加した知識」だけを表示。週が替わると先週分はホームから消える（データは残る）。
  getWeekStart() が直近の木曜0:00を計算。検索・タグ時や「すべて見る」時は
  週の絞り込みを外して全件対象（showAllMode で管理）。今週0件なら案内メッセージ。

### 案4：ランダム閲覧画面（✅ 完了）
- 何を：登録済みからランダムに最大10件を「タイトルのみ」で表示する画面を追加。
  通知ではない「能動的な再会」で、忘れかけた知識に出会える。
- 実装：#view-random を追加。loadRandomList が Fisher–Yates でシャッフルして
  最大10件を表示。「ランダムに見る」「次のランダム」「← 一覧に戻る」。未ログインでも使える。

### 画像機能（✅ 完了・2026-06-22）
